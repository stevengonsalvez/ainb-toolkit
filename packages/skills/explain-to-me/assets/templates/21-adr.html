Architecture Decision Record

ADR-0023 — Move session state to event-sourced store

Date2026-04-12 Authors@stevie, @lin, @marc DecidersPlatform Eng Accepted
01

Context

Session state today lives in a single Postgres row updated in place on every tool call. We are seeing three concrete pains: lost-update races between background drainers and live writers, expensive replay (we copy the whole row to diff against a snapshot), and increasingly tangled migrations because the row is a JSON blob.

Forces in tension
  • Replay must be cheap — agents reconstruct state on every cold start
  • Schema changes need to ship weekly without coordinated downtime
  • Audit trail required by compliance (who changed what, when)
  • p99 write latency must stay under 25ms
  • Operating two stores doubles paging surface
02

Options considered

Three approaches were prototyped against the same workload (10k concurrent sessions, mixed read/write).

Option A

Keep blob row, add row-level lock

Add SELECT…FOR UPDATE around writes. Minimal change to data model.

  • Smallest diff
  • No new infra
  • Doesn't fix replay cost
  • Locks serialise drainers
Effortlow
Fitpoor
Risklow
Option BCHOSEN

Append-only event log + projected view

Writers append events. A materialised view projects current state and is rebuildable from the log.

  • No update conflicts
  • Free audit trail
  • Cheap replay
  • Two storage shapes
  • Projection lag
Effortmed
Fitstrong
Riskmed
Option C

Move to dedicated CRDT store

Replace the row with a CRDT-backed key/value store. Mergeable by design.

  • No locks ever
  • Offline-capable
  • New ops surface
  • Hard to query ad-hoc
  • Team unfamiliar
Efforthigh
Fitok
Riskhigh
03

Decision

B
We will adopt
Append-only event log with a projected materialised view
Option B is the only one that fixes both lost-update races and replay cost in the same change, without forcing the team onto unfamiliar infra. The projection-lag concern is bounded (sub-second under observed load) and recoverable from the log if it ever drifts.
04

Resulting architecture

End-state once the migration completes. Writers only touch the log; readers only touch the view.

Solid arrows = synchronous · dashed olive = async stream · dashed rust = cold-path replay
05

Consequences

Positive

  • Lost-update races become structurally impossible — log is append-only
  • Replay reduces from 480ms p99 to ~35ms p99 (measured on the prototype)
  • Audit trail is the source of truth, not a separate denormalised copy
  • Schema evolves via new event types, no online migration needed

Negative

  • Two storage shapes to operate: log + projection
  • Engineers must learn event-versioning discipline (no in-place edits)
  • Projection-lag windows (sub-second) require careful read-after-write care

Neutral

  • Storage cost roughly flat (logs are smaller per event but more numerous)
  • Existing query tooling continues to work against the projection view
